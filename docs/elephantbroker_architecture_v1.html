ElephantBroker: A Knowledge-Grounded Cognitive Runtime for Trustworthy AI Agents

Unified Architecture for Memory Plugin + ContextEngine Plugin on Cognee
Authors: Cristian Lupascu, Alexandru Lupascu Revision: v1 Date: 2026-04-03 Paper: arXiv:2603.25097

1. Executive Summary

Current agent memory systems exhibit three fundamental shortcomings: (1) no relational structure beyond flat vector stores, (2) no provenance or trustworthiness tracking on stored memories, and (3) retrieval disconnected from the agent's current goals.

ElephantBroker is a single Cognitive Runtime that addresses all three through a unified architecture with:

  • A Memory Plugin as the durable knowledge, actor, procedure, evidence, and consolidation surface
  • A ContextEngine Plugin as the working-set, assembly, compaction, guard, and subagent surface
  • A shared Cognee-backed knowledge plane underneath both — Python-native, leveraging Cognee's DataPoint model, graph + vector search, and custom task pipelines
  • A shared policy/profile layer governing retrieval, assembly, compaction, verification, and red-line enforcement
  • An AI Firewall that transforms advisory guard verdicts into physically enforceable tool-call interception across three defense layers

This design gives three things at once:

  1. Standalone viability — The Memory Plugin can be sold and used alone
  2. Composable context intelligence — The ContextEngine can run alone with reduced durability
  3. Best-in-class full system behavior — Memory + Context together behave like one coherent operating system for agents
The cognitive loop: store → retrieve → score → compose → protect → enforce → learn. Every step is goal-aware, evidence-grounded, profile-tuned, and gateway-isolated.

2. Principal Contributions

ElephantBroker makes eight principal contributions to the agent memory and context management problem:

  1. Hybrid five-source retrieval pipeline — concurrent structural graph queries, lexical matching, semantic vector search, ego-net graph expansion, and artifact retrieval, merged with configurable per-source weights and actor-level isolation policies
  2. Eleven-dimension competitive scoring engine — evaluates candidates on turn relevance, session/global goal relevance, recency, successful use, confidence with verification multipliers, evidence strength, novelty, redundancy penalty, contradiction penalty, and token cost penalty. Budget-constrained greedy selection with a (1 − 1/e) submodularity guarantee
  3. Four-state evidence verification model — UNVERIFIED → SELF-SUPPORTED → TOOL-SUPPORTED → SUPERVISOR-VERIFIED (plus REJECTED from any state). Verification status feeds graded confidence multipliers into scoring
  4. Five-stage context lifecycle — bootstrap, ingest, assemble, compact, afterTurn. Manages profile resolution, four-block assembly across two prompt surfaces, two-stage compaction, six-signal successful-use detection, and subagent spawn/end hooks
  5. Six-layer cheap-first guard pipeline — autonomy domain classification, static rule matching, semantic matching, structural validation, forced constraint reinjection, and optional LLM escalation
  6. Nine-stage consolidation engine — cluster near-duplicates, canonicalize, strengthen useful facts, decay unused, prune bad auto-recall, promote episodic to semantic, refine procedures from patterns, identify verification gaps, recompute salience priors. Seven of nine stages require zero LLM calls
  7. AI Firewall — transforms advisory guard verdicts into physically enforceable tool-call interception across three defense layers (agent hook, proxy guardrails, SDK hooks), with six-tier cost-ordered safety scanning and knowledge base contamination prevention
  8. Authority-based multi-organization identity model — numeric permission hierarchy with four tiers, eleven configurable authority rules with matching-exempt semantics, organizations and teams as first-class graph nodes

3. Design Principles

3.1 One runtime, two plugin surfaces

The plugins are product surfaces, not independent brains. They share schema, IDs, cache keys, policy resolution, profile behavior, actor registry, stats, evidence references, goal model, and working-set semantics. The runtime is written in Python; only thin plugin adapters exposed to host frameworks live in TypeScript.

3.2 Working memory is first-class

The system distinguishes clearly between chat history, durable memory, working set, and prompt payload. The prompt is an output artifact, not the source of truth.

3.3 Goals are first-class

The system tracks global goals, organization goals, session goals, task goals, and subagent goals. Every lifecycle step knows which goal level it serves. Goals shape scoring, compaction, retrieval, guard evaluation, and consolidation.

3.4 Actors are first-class

Do not model the world as "one human user + one assistant." Model human coordinators, peer agents, supervisor agents, worker agents, external humans, and services/tools/system actors. Every memory entry tracks source/target actors, affected goals, scope, and provenance.

3.5 Claims require evidence

Any statement of action, completion, or compliance is optionally backed by evidence references with a four-state verification model. Verification status feeds graded confidence multipliers (0.5, 0.7, 0.9, 1.0) into scoring, creating a structural bias toward trustworthy information.

3.6 Cheap safety before expensive safety

Red-line enforcement starts with autonomy domain classification → static rule matching → semantic similarity → structural validators → forced reinjection → LLM escalation. Six layers, each cheaper than the next. Most violations are caught before any LLM call is needed.

3.7 Compaction is continuous and goal-aware

Compaction is not an occasional emergency response. It is a normal lifecycle activity with the current goal in mind. Two-stage: classify (zero LLM) then summarize (one LLM call).

3.8 Gateway-first isolation

Every entity carries a gateway_id. All Redis keys, Neo4j queries, Cognee datasets, Prometheus metrics, and trace events are gateway-scoped. No single-tenant assumptions anywhere in the stack.

4. Four-Layer Architecture

The system is organized into four layers. Layer A handles agent integration via thin TypeScript plugins. Layer B is the Python cognitive runtime with 17 modules. Layer C is the Cognee SDK knowledge plane. Layer D is the infrastructure plane.

12. Hybrid Five-Source Retrieval

The retrieval pipeline is formalized as:

f(α) = σ(ρ(φ(α)))

where α = query string, φ = multi-source search (recall), ρ = four-stage reranker (precision), σ = eleven-dimension scoring + budget selection (value optimization).

12.1 Five sources dispatched concurrently

SourceMethodDefault WeightDepends On
StructuralNeo4j Cypher: session_key, actor, memory_class, scope filterswstruct = 0.4Nothing
KeywordCognee CHUNKS_LEXICAL (BM25-like)wkw = 0.3cognee.cognify()
SemanticCognee CHUNKS or direct vector with payload filters under STRICT isolationwsem = 0.5cognee.cognify()
Graph expansionCognee GRAPH_COMPLETION — ego-net traversal to configurable depthwgraph = 0.2add_data_points() vectors; richer after cognify()
ArtifactVector search over dedicated artifact collectionswart = 0.5Artifact ingest

12.2 Merge strategy

Multiply raw score by source weight, deduplicate by fact ID retaining maximum weighted score, sort descending, cap at configurable maximum. Each source has enable/disable, fetch_k, and weight — tunable per profile via RetrievalPolicy.

12.3 Graceful degradation

Five-level hierarchy: source-level skip, method-level degradation, pipeline-level degradation, operation-level degradation, edge-level degradation. No single backend failure blocks the pipeline. Under STRICT isolation, semantic source falls back to direct vector search with payload filters.

12.4 Graph modes by profile

LOCAL — execution, coding, narrow task continuation. HYBRID — general default. GLOBAL — research, strategy, cross-project synthesis. Graph depth: 1 (coding), 2 (managerial, personal), 3 (research).

14. Working Set Manager

This is the center of the architecture. Of everything the system knows, what deserves scarce prompt tokens right now?

14.1 Winner-takes-budget model

A fact is not injected because it was recalled — it is injected because it won a budgeted competition. The WorkingSetManager generates candidates from all memory classes via the 5-source retrieval pipeline, scores each on 11 dimensions, and selects winners under the profile's token budget with a (1 − 1/e) ≈ 0.63 approximation guarantee (submodular optimization).

14.2 Selection algorithm

  1. Pre-allocate mandatory items unconditionally: constraints, goals with blockers, procedures requiring proof
  2. Pass 1: Compute 9 independent scoring dimensions for all remaining candidates
  3. Sort by Pass 1 weighted sum, descending
  4. Greedy selection with Pass 2: For each candidate in order, compute 2 interaction-dependent dimensions (redundancy + contradiction) against already-selected items. Include if total score > 0 and budget permits

14.3 Candidate sources

must-inject constraints (always win) · session goals with blockers · procedures requiring proof · active facts from 5-source retrieval · evidence summaries · artifact summaries · compact state entries

15.4 Tuning mechanism

The ScoringTuner implements a feedback-driven tuning loop during consolidation. It computes Spearman correlations between each scoring dimension and the successful-use signal, then applies EMA-smoothed weight adjustments capped at ±5% per cycle to prevent oscillation. Tuning deltas are persisted in SQLite, scoped per profile/org/gateway. Weight resolution chain: base preset → profile → org override → tuning delta.

16. Context Lifecycle

16.1 bootstrap

Resolves profile policy (immutable for session lifetime). Registers agent actor if new. Loads session goals from Redis. Creates session context record in cache. Loads compacted state snapshot. Seeds working set with must-have constraints and goals.

16.2 ingest

Turn ingest pipeline processes messages through ten sequential tasks: actor resolution, recent fact loading, LLM extraction with structured schema, supersession/contradiction detection, memory classification, batch embedding, dedup at 0.95, storage via add_data_points(), Cognee enrichment. Artifact ingest handles tool outputs with SHA-256 dedup. Procedure ingest extracts multi-step workflows.

16.3 assemble

Working set manager produces scored snapshot. Guard engine runs preflight check. Context assembler distributes items across four blocks on two prompt surfaces:

SurfaceBlockContent
Surface A (system prompt addition — highest attention)Block 1: systemPromptAdditionConstraints, guard reinjections, proof-required procedure instructions
Block 2: prependSystemContextGoals with blockers highlighted
Surface B (prepended context)Block 3: prependContextScored working set, ordered by memory class (POLICY → PROCEDURAL → SEMANTIC → EPISODIC) then by score descending
Block 4: appendSystemContextEvidence references

16.4 compact

Two-stage compaction. Stage 1 (zero LLM cost): classifies each message as PRESERVE, COMPRESS, or DROP. Stage 2 (one LLM call): summarizes the COMPRESS bucket into 1–3 paragraphs with goal-aware prompts. Cadence thresholds vary by profile aggressiveness (1.3x to 4x target token count).

16.5 afterTurn

Six-signal successful-use detection:

SignalMethodWeight
S1: Direct QuoteSubstring match of injected fact in LLM response0.95
S2: Tool CorrelationTool call arguments match injected fact content0.70
S3: Goal MentionResponse references a goal the fact was tagged for0.50
S4: Emergent BehaviorResponse shows action consistent with injected constraint0.40
S5: ContinuationFollow-up turn builds on injected fact0.35
S6: Ignored Penalty3+ turns with zero signals — negative marker−0.30

Running Jaccard coefficient > 0.3 triggers successful-use increment. Smart goal injection cadence: goals re-injected only when at least one of five conditions holds (goal status changed, new blocker, progress recorded, N turns since last injection, explicit user request).

16.6 Subagent support

prepareSubagentSpawn: filters parent working set by child goal relevance with 2x budget allocation. onSubagentEnded: merges child facts into parent, updates parent goals, verifies declared completion where required.

17. Six-Layer Guard Pipeline

Cheap-first pipeline: Each layer is faster and cheaper than the next. Most violations are caught before any LLM call. Five policy presets (coding, research, managerial, worker, personal_assistant) control strictness per layer.

Layer 0 — Autonomy domain classification

Classifies agent actions into ten domains: FINANCIAL, DATA_ACCESS, COMMUNICATION, CODE_CHANGE, SCOPE_CHANGE, RESOURCE, INFORMATION_SHARE, DELEGATION, RECORD_MUTATION, UNCATEGORIZED.

Four-point autonomy scale per domain: AUTONOMOUS, INFORM, APPROVE_FIRST, HARD_STOP. Floor composition: final = max(autonomy_floor, safety_result) — safety can only escalate, never relax.

Layer 1 — Static rule matching

Four-source rule registry: built-in rules (12 default patterns), profile-specific rules, procedure-bound rules (loaded dynamically from active procedures' guard bindings), operator-defined custom rules. Match by keyword, phrase, regex, or tool target patterns. Cost: microseconds.

Layer 2 — Semantic matching

BM25 scoring (block threshold 8.0, warn threshold 5.0) and cosine similarity (threshold 0.80) against known-bad exemplars. Per-profile strictness modifier: loose = 1.5x thresholds, strict = 0.7x thresholds. Cost: milliseconds.

Layer 3 — Structural validation

Graph queries for approval tokens, evidence existence, and confirmation flags. Disabled under loose strictness. Cost: milliseconds.

Layer 4 — Forced constraint reinjection

Always executes. Collects all triggered constraints and formats them as mandatory working-set items for Block 1 of context assembly. The agent sees all applicable constraints even when the verdict is pass or warn.

Layer 5 — Optional LLM escalation

Narrow safety-focused prompt for ambiguous cases. Only fires under strict profiles or when explicitly enabled. Cost: one LLM call.

Guard outputs and escalation

Five result types: pass, warn, block, require_approval, require_evidence. Near-miss escalation: three warn results within five turns auto-tightens strictness (medium → strict → LLM layer enabled).

Procedures-Guards-Evidence triangle

The "Triangle of Accountability": procedure activation loads guard bindings (connection 1). Guard violation demands evidence from agent (connection 2). Procedure completion check validates all proof requirements have verified claims (connection 3).

18. AI Firewall

The guard pipeline (Section 17) provides advisory enforcement — it shapes context assembly but cannot physically prevent tool calls. The AI Firewall adds physical enforcement, forming a "belt-and-suspenders" dual-enforcement architecture.

18.1 Three physical enforcement layers

19. Evidence Verification Model

19.1 Four-state machine

Claims progress through a verification state machine with graded confidence multipliers (mv) that feed directly into the scoring engine:

StatemvTransition Trigger
UNVERIFIED0.5Initial state for all new claims
SELF-SUPPORTED0.7Any evidence attached (e.g., reference to supporting fact)
TOOL-SUPPORTED0.9Tool output evidence provided (e.g., API call result, diff hash)
SUPERVISOR-VERIFIED1.0Human supervisor explicit sign-off
REJECTED—Reachable from any state. Mandatory reason string + rejector actor ID

19.2 Completion contract

For procedures with proof requirements, completion is valid only if: all required steps executed, all required evidence attached, all red-line checks passed, and any required approval exists. The validate_completion() method enforces this contract.

19.3 HITL approval queue

Redis-backed approval queue for require_approval guard verdicts. A separate HITL middleware service (port 8421) provides the approval/rejection interface. Auto-goals created for pending approvals to surface them in the working set.

20. Traceability & Observability

Every relevant event creates an append-only trace event from a vocabulary of 47 event types. Events reference artifact IDs, chunk refs, claim IDs, procedure IDs, goal IDs, and actors involved.

20.1 Trace API

Six endpoints: POST /trace/query (filtered search), GET /trace/sessions (list), GET /trace/session/{id}/timeline, GET /trace/session/{id}/summary, GET /trace/event-types. All gateway-scoped.

20.2 Observability stack

  • Traces: @traced decorator auto-spans; session_key/gateway_id propagated via W3C trace context
  • Logs: GatewayLoggerAdapter prepends [gateway_id][agent_key] to all messages
  • Metrics: 49 Prometheus counters/gauges via StatsEngine (retrieved, stored, dedup, guard hits, compaction, etc.)
  • Analytics: TraceLedger → OTEL Collector → ClickHouse for cross-session analytics. Grafana dashboards.

21. Compaction Architecture

21.1 Goal-aware compaction preserves

Current session goal state, blockers, decisions made, unresolved questions, critical constraints, evidence refs needed later, important actor relationships.

21.2 Two-stage classification

Stage 1 (zero LLM): Rule-based classification — PRESERVE (goals, constraints, open blockers), COMPRESS (explanations, repeated injections, stale context), DROP (phatic text, superseded artifacts, restated information). Stage 2 (one LLM call): Summarize the COMPRESS bucket into 1–3 paragraphs with goal-aware prompts.

21.3 Post-compaction feedback

Items well-represented in compact state get their novelty score zeroed in subsequent working-set competitions. Successful-use signals (S1–S6) track which compacted items the LLM actually used.

22. Consolidation / "Sleep" Pipeline — 9 Stages

Analogous to memory consolidation during sleep. Runs during idle periods with a Redis distributed lock (1-hour timeout). Gateway-scoped. Seven of nine stages require zero LLM calls.

Stage 1 — Cluster near-duplicates

Cosine similarity > 0.92 threshold via Cognee vector search. Pure vector math + union-find clustering. Zero LLM.

Stage 2 — Canonicalize

Select highest-confidence, most-cited version as canonical. Merge provenance from all versions. LLM only for ambiguous clusters (<5% of cases).

Stage 3 — Strengthen useful facts

Confidence boost: c'(f) = min(c(f) + s(f)/u(f) × 0.3, 1.0) where s(f) = successful use count, u(f) = total use count. Zero LLM.

Stage 4 — Decay unused facts

Two modes. Recalled but unused: c' = c × 0.9tr (tr = recall turns). Never recalled: c' = c × 0.95td (td = days since creation). Decay rates by scope: SESSION hourly, ACTOR daily, TEAM/ORG weekly, GLOBAL monthly. Facts below 0.1 are archived. Zero LLM.

Stage 5 — Prune bad autorecall

Facts auto-recalled 5+ times with zero successful uses are blacklisted from automatic injection. Remain searchable. Zero LLM.

Stage 6 — Promote episodic → semantic

Recurring episodic facts (3+ sessions, embedding similarity > 0.95 cross-session) promoted to semantic class. Memory class and scope co-promote. Zero LLM.

Stage 7 — Refine procedures from patterns

Trace ledger analysis for repeated multi-step patterns (3+ occurrences). LLM generates draft ProcedureDefinition. Goes to review queue — never auto-activated. 1 LLM call per pattern.

Stage 8 — Identify verification gaps

Claims compared against procedure proof requirements. Missing evidence flagged for supervisor sampling. Zero LLM.

Stage 9 — Recompute salience priors

EMA adjusts per-profile scoring weights from cumulative use data. Spearman correlation between dimensions and success signal. Capped at ±5% per cycle. Persisted in SQLite. Zero LLM.

Consolidation outputs

Most recalled · Most useful · Most wasted-prompt · Procedures correlated with success · Claims frequently rejected · Artifacts repeatedly reused · Safe to archive · Suggested procedures · Verification gaps · Weight tuning deltas

23. Procedural Memory

23.1 Capabilities

Workflows, checklists, escalation paths, red-line guard bindings, proof requirements per step, role-specific variants, approval requirements, retry/fallback patterns, supervisor review hooks.

23.2 Activation modes

Manual selection · Actor/profile default · Trigger-word based · Task classifier based · Goal-bound · Supervisor-forced.

23.3 Guard bindings

Procedures declare guard bindings that activate when the procedure activates. This dynamically extends the guard engine's rule set for the duration of the procedure, creating the Procedures-Guards-Evidence triangle described in Section 17.

23.4 Procedure → Context interaction

A procedure can influence context assembly by: adding must-inject constraints (Block 1), reserving budget for required evidence, forcing a completion schema, altering compact/preserve policy, changing subagent packet structure.

24. Subagent Packet Management

24.1 Spawn mechanism

The prepareSubagentSpawn hook filters the parent's working set by child goal relevance, allocating 2x budget. Depth is configurable (default: max 3 levels of nesting). Subagent spawns are bounded events — typically 0–5 per session.

24.2 Packet contents

Subgoal, local constraints, relevant procedures, essential actor relationships, necessary evidence references, bounded artifact snippets. Not the full parent context.

24.3 Return merge

onSubagentEnded: ingest returned claims/artifacts/evidence, score relevance to parent goal, verify declared completion, update trace, promote accepted facts, compact large outputs.

27. Cognee Knowledge Plane Integration

27.1 Storage pattern

All DataPoint subclasses are stored via add_data_points(). This automatically: stores all custom fields as Neo4j node properties via MERGE by ID, embeds and indexes index_fields in Qdrant (e.g., FactDataPoint_text collection), and supports upsert on same ID. Cognee's GRAPH_COMPLETION search auto-discovers DataPoint subclasses via get_all_subclasses(DataPoint).

27.2 Ingest pipelines

  • turn_ingest — messages → LLM fact extraction → classification → supersession detection → dedup → storage
  • artifact_ingest — tool outputs → SHA-256 dedup → LLM extraction → summary → storage
  • procedure_ingest — completion logs → procedural knowledge extraction

27.3 Search pattern (hybrid)

  • cognee.search(GRAPH_COMPLETION, only_context=True) — semantic/graph discovery
  • cognee.search(CHUNKS) — semantic chunk search
  • cognee.search(CHUNKS_LEXICAL) — BM25-like keyword search
  • GraphAdapter.query_cypher() — structural hits by scope, actor, session_key
  • VectorAdapter.search_similar() — filtered vector search on Cognee-managed collections

27.4 Dataset scoping

All cognee.add() calls use dataset_name=f"{gateway_id}__{base}" for gateway-scoped datasets. cognee.cognify() runs Cognee's full pipeline: chunking, entity extraction, relationship extraction, triplet embedding — making GRAPH_COMPLETION search work.
